# Lessons Learned - 实验教训总结

## 🔴 重大教训 (Must Read)

### Lesson 001: 代码任务禁止自反射
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-07 |
| **版本** | v9.0 |
| **问题** | 对代码任务应用自反思策略导致 Gen 分数灾难性失败 |
| **影响** | Gen 从 68.6 降到 20 分以下 |
| **教训** | 代码任务必须一次完成，自反思对代码任务无效 |
| **行动项** | 策略中对代码任务设置 `reflect: none` |

### Lesson 002: 6000 tokens 边际递减
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-07 |
| **版本** | v32.0 |
| **问题** | 增加 tokens 到 6000 导致 Core 分数下降 7.6 分 |
| **影响** | Core 从 79.2 降到 71.6 |
| **教训** | 5000 tokens 是临界点，增加反而降低质量 |
| **行动项** | 固定 research_tokens=5000, code_tokens=5000 |

### Lesson 003: 策略定义但从未应用
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-08 |
| **版本** | v31_0 演化版 |
| **问题** | harness_evolution.py 定义了策略池但 `run_harness()` 函数直接复制模板，完全忽略策略参数 |
| **影响** | 所有"演进"实际运行的都是相同的 v31_0 代码 |
| **教训** | 必须验证策略是否真正应用，不能假设有效 |
| **行动项** | 添加策略验证步骤：检查生成代码中的参数值 |

### Lesson 004: 假分数污染实验历史
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-09 |
| **版本** | evolution_loop.py |
| **问题** | evolution_loop.py 返回 `random.uniform(-5, 10)` 的假分数 |
| **影响** | 无法区分真实实验结果和模拟结果 |
| **教训** | 必须验证数据来源，绝不接受未确认的分数 |
| **行动项** | 重命名为 _MOCK.py，标记为不可信 |

### Lesson 005: API key 硬编码暴露
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-09 |
| **版本** | 多个 harness 版本 |
| **问题** | API key 直接写在源代码中，进入 git 历史 |
| **影响** | 安全风险，key 可能被滥用 |
| **教训** | 所有凭证必须通过环境变量获取 |
| **行动项** | 使用 `os.environ.get("MINIMAX_API_KEY")` |

### Lesson 006: 双路径写入导致状态不一致
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-09 |
| **版本** | v31_0 + evolution |
| **问题** | `harness_v31_0.py` 和 `harness_evolution.py` 都可以写入 state.json |
| **影响** | 竞态条件，状态可能不一致 |
| **教训** | 单一权威写入点，所有其他写入必须禁用 |
| **行动项** | 只允许 evolution.py 写入 state.json |

### Lesson 007: Agent 偷懒与幻觉串通
| 属性 | 值 |
|------|-----|
| **日期** | 2026-04-09 |
| **问题** | 执行 Agent 可能偷懒产生低质量输出，检查 Agent 可能幻觉生成虚假高分 |
| **影响** | 系统性欺诈，误判为成功 |
| **教训** | 必须三重验证，永不依赖单一来源 |
| **行动项** | 实现 ANTI_FRAUD.md 协议 |

## 🟡 中等教训

### Lesson 008: 字符串匹配代码生成脆弱
| 属性 | 值 |
|------|-----|
| **问题** | `apply_strategy_to_code()` 使用硬编码字符串匹配，容易因格式变化失效 |
| **影响** | 策略参数静默失败 |
| **教训** | 使用 AST 操作或模板引擎，不用字符串匹配 |
| **行动项** | 考虑 Jinja2 模板 |

### Lesson 009: 无资源限制导致潜在 OOM
| 属性 | 值 |
|------|-----|
| **问题** | resource_limiter.py 存在但从未被 import/使用 |
| **影响** | 可能耗尽内存 |
| **教训** | 定义了机制必须实际使用 |
| **行动项** | 在所有 harness 启动时调用 `apply_all()` |

### Lesson 010: checkpoint 格式不一致
| 属性 | 值 |
|------|-----|
| **问题** | checkpoint 有时保存完整数据，有时只有简化版 |
| **影响** | 无法可靠恢复 |
| **教训** | 统一 checkpoint 格式 |
| **行动项** | 定义 checkpoint schema 并验证 |

## 🔬 实验具体分析 (Experiment RCA)

### v31_1 (69.25) vs v31_0 (76.22) - 倒退 7 分

| 指标 | v31_0 | v31_1 | 变化 |
|------|--------|--------|------|
| Core | 79.2 | 70.0 | -9.2 |
| Gen | 81.0 | 72.0 | -9.0 |
| Actionability | 4.13 | ~3.8 | -0.3 |

**可能原因**：
- 策略参数变化导致输出质量下降
- 需要检查 v31_1 使用的具体参数

**教训**：即使是"优化"也可能导致回归，必须有 AB 测试

### v40_0 (51.74) vs v31_0 (76.22) - 暴跌 25 分

| 指标 | v31_0 | v40_0 | 变化 |
|------|--------|--------|------|
| Core | 79.2 | ~61.2 | -18 |
| Gen | 81.0 | ~50.75 | -30.25 |

**可能原因**：
- "FAST mode" 3000 tokens 太低
- MAX-1 策略减少候选次数
- 输出长度和质量显著下降

**教训**：
- Tokens 过低会严重影响质量
- 3000 tokens 是临界值以下

### v33 (73.44) vs v32 (72.22) vs v38 (72.16)

| 版本 | 策略 | 分数 |
|------|------|------|
| v33 | MAX-3 | 73.44 |
| v32 | 6000 tokens | 72.22 |
| v38 | ? | 72.16 |

**观察**：
- MAX-3 (多次运行取最优) 表现最好
- 6000 tokens 相比 5000 没有提升
- v38 继承了 v32 的参数但分数相似

**教训**：
- MAX 策略比单纯增加 tokens 更有效
- 需要记录每个版本的具体参数

### evo_001 运行问题

**观察到的问题**：
- API key 未设置导致失败
- 51.74 分数是在 API 失败时产生的虚假分数
- state.json 没有正确更新

**教训**：
- 必须先验证环境再运行
- 虚假分数可能污染实验历史
