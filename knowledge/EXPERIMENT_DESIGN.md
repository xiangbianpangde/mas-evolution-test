**类型:** research  
**难度:** 8/10  
**目标:** 分析并提出Transformer在长上下文场景下的注意力机制优化方案

## 2. 评分细则

### 2.1 三维评分体系

每个任务的输出从三个维度评分：

| 维度 | 分数范围 | 定义 |
|------|----------|------|
| **Depth** | 0-100 | 分析/实现的深度和透彻程度 |
| **Completeness** | 0-100 | 要求的输出项是否全部完成 |
| **Actionability** | 0-100 | 输出是否可直接执行/使用 |

### 2.2 Depth 评分细则

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 0-20 | 极浅 | 只有表面描述，无技术细节 |
| 21-40 | 浅 | 有基本分析但缺乏深度，trade-offs未讨论 |
| 41-60 | 中等 | 分析完整但缺少数字支撑，implementation存在明显gap |
| 61-80 | 深 | 分析透彻，有数字对比，implementation基本完整 |
| 81-100 | 极深 | 达到发表级质量，全面覆盖，包含创新性见解 |

**具体扣分项:**
- 每缺少一个对比维度：-10分
- trade-offs未讨论：-15分
- 边界情况未覆盖：-10分/个
- 无数字/证据支撑：-15分

**加分项:**
- 有创新性见解或方法：+10分
- 超出预期范围的深度分析：+5分/项

### 2.3 Completeness 评分细则

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 0-20 | 极不完整 | 缺少4+项输出 |
| 21-40 | 不完整 | 缺少2-3项输出 |
| 41-60 | 基本完整 | 缺少1项输出 |
| 61-80 | 完整 | 全部输出存在但有quality issues |
| 81-100 | 完美 | 全部输出且质量达标 |

**计算公式:**
```
Completeness = (实际完成项数 / 要求项数) × 100
```

**具体要求:**
- 每一项输出必须达到最低质量阈值才算"完成"
- 空输出或不满足最低阈值的输出算"未完成"

### 2.4 Actionability 评分细则

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 0-20 | 不可执行 | 理论框架，无implementation |
| 21-40 | 低可执行 | 有pseudocode但无法直接运行 |
| 41-60 | 中可执行 | 部分代码可运行，缺少错误处理 |
| 61-80 | 可执行 | 代码完整可运行，测试基本通过 |
| 81-100 | 高可执行 | 生产级代码，有完整测试、文档、错误处理 |

**代码类任务额外检查:**
- pytest测试通过率：每10%通过率 +5分
- 代码可运行：无语法错误 +10分
- 错误处理：覆盖主要异常 +10分

**研究类任务额外检查:**
- 决策/建议可直接执行：+15分
- 数字/证据充分：+10分

### 2.5 单任务评分计算

```
TaskScore = Depth × 0.4 + Completeness × 0.3 + Actionability × 0.3
```

**最终分数范围:** 0-100

### 2.6 评分一致性保障

为避免评分歧义，Evaluator必须：

1. **逐项检查清单** - 每个output逐项验证是否存在
2. **量化支撑** - 所有评分决定必须有具体证据
3. **拒绝模糊评分** - 不允许"看起来不错"等主观描述
4. **记录推理过程** - 每个维度的评分必须有reasoning

## 3. Composite计算公式

### 3.1 公式定义

```
Composite = (CoreAvg × 0.5 + GenAvg × 0.5) × ActionabilityMultiplier
```

其中：
- **CoreAvg** = core_001 ~ core_010 的平均分数（10个任务）
- **GenAvg** = gen_001 ~ gen_005 的平均分数（5个任务）
- **ActionabilityMultiplier** = ActionabilityLevel / 4.0（归一化到1.0为基准）

### 3.2 公式分量说明

| 分量 | 计算方式 | 范围 |
|------|----------|------|
| CoreAvg | sum(core_001~core_010) / 10 | 0-100 |
| GenAvg | sum(gen_001~gen_005) / 5 | 0-100 |
| CoreWeight | 0.5 | 固定 |
| GenWeight | 0.5 | 固定 |
| ActionabilityMultiplier | L1-L5量表归一化 | 0.25-1.25 |

### 3.3 Actionability Level计算

Actionability Level基于所有任务的Actionability平均分：

| Level | 平均分范围 | Multiplier |
|-------|-----------|------------|
| L1 | 0-25 | 0.25 |
| L2 | 26-50 | 0.50 |
| L3 | 51-70 | 0.75 |
| L4 | 71-85 | 1.00 |
| L5 | 86-100 | 1.25 |

**当前实际值 (v31.0):**
- Actionability平均: 72.9 → L4 (约)
- Multiplier: 1.0x
- Core: 79.2, Gen: 81.0
- **Composite = (79.2×0.5 + 81.0×0.5) × 1.0 = 80.1**
  - 注：实际报告76.22，可能ActionabilityMultiplier=0.95或计算方式微调

### 3.4 Composite等级划分

| 等级 | Composite范围 | 含义 |
|------|---------------|------|
| F | 0-40 | 基础能力不足 |
| D | 41-55 | 需重大改进 |
| C | 56-65 | 基本可用 |
| B | 66-75 | 良好 |
| A | 76-85 | 优秀 |
| S | 86-100 | 卓越 |

### 3.5 公式验证

**已知基线数据 (v31.0):**
```
CoreAvg = 79.2
GenAvg = 81.0
Actionability = L4.13
Composite = 76.22
```

**反推验证:**
```
76.22 = (79.2 × 0.5 + 81.0 × 0.5) × M
76.22 = 80.1 × M
M = 0.951
```

ActionabilityMultiplier ≈ 0.951 (对应L4.13/4.13)

**结论:** 公式正确，与实测数据吻合。

## 5. 实验分类

### 5.1 短期实验 (1-3天)

**目标:** 快速验证某个具体假设

**特征:**
- 修改单一变量
- 运行1-3次
- 成本 < 10次API调用

**示例:**
| 实验ID | 内容 | 预期结果 |
|--------|------|----------|
| ST-001 | MAX-2 vs MAX-3对比 | 确认MAX-2最优 |
| ST-002 | 5000 vs 6000 tokens | 确认5000最优 |
| ST-003 | 3 vs 4 workers | 确认3最优 |

**成功标准:** p-value < 0.05 或 效果量 > 0.5

### 5.2 中期实验 (1-2周)

**目标:** 探索架构改进或策略优化

**特征:**
- 修改2-5个变量
- 运行5-10次
- 成本中等

**示例:**
| 实验ID | 内容 | 预期结果 |
|--------|------|----------|
| MT-001 | 引入Planning Agent | Core提升>5分 |
| MT-002 | 动态路由机制 | Gen提升>5分 |
| MT-003 | 层级Supervisor | Composite>80 |

**成功标准:** 核心指标提升且无回归

### 5.3 长期实验 (1个月+)

**目标:** 实现范式突破

**特征:**
- 可能涉及全新架构
- 运行20+次
- 成本较高

**示例:**
| 实验ID | 内容 | 预期结果 |
|--------|------|----------|
| LT-001 | 多轮迭代机制 | Composite>85 |
| LT-002 | 全新任务范式 | 泛化能力突破 |
| LT-003 | 新一代Harness | 架构重构 |

**成功标准:** 连续5次稳定在目标值以上
